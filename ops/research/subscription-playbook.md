# 订阅制预案(未来 Pro 化时启用 · 2026-07-23)

> 现状:$49 一次性 Payment Link(零后端)是当前正确的最小方案。**一旦上订阅制,本档全部条目变必修课。**
> 第一个架构决定要提前拍板:静态 SPA + GitHub Pages 无后端 —— 届时要么加最小 serverless webhook 接收端,要么用 Creem 类 MoR(商家记录方)。

## 一、Webhook 五事件标准配置
监听:`invoice.paid` · `checkout.session.completed` · `customer.subscription.created / updated / deleted`。
本地测试:Stripe CLI 转发;失败事件 Dashboard 手动重试。
测试卡:4242…(成功)· 4000 0000 0000 3220(3DS)· 4000 0000 0000 9995(余额不足)。

## 二、续费逻辑测试技巧
把订阅计费周期临时设 **1 天**,次日即验证"续费后权益发放"链路(自动续费类 bug 最难测,这是唯一低成本复现法)。

## 三、权益发放 cron 兜底(血泪案例)
年费会员按月发权益**必须**每日定时任务兜底(cron-job.org 免费,ADVANCED 配 Basic Auth 防接口裸奔)。反例:漏配后 3 个年费用户没拿到权益,靠投诉才发现。
—— 与执行手册避坑 12"管道建好≠水会流"同一铁律的收款版。

## 四、防薅羊毛三招
1. Google 登录挡不住多账号(一人几十个 Google 号薅试用是常态)。
2. 同 IP 短时多次注册 → 递减赠送额度;可按地区差异化赠送。
3. 上游 API key 设每日消费上限 = 历史峰值 2 倍,密钥只放服务端。
订阅制按周期一次性发权益,比按次计费简单一个量级。

## 五、积分/额度运营细则
- 充值档预留 ≥2 倍日常用量;支持自动充值。
- 每日消耗上限动态调整为历史峰值 2 倍(防单点爆刷)。
- 渐进结果展示可作付费墙形态:先看部分结果(模糊报告/水印/只给汇总)再引导付费——注意与"游客模式提转化"的平衡,别在"啊哈时刻"之前设卡。

## 六、涨价五法(定价演进期用)
1. 新用户涨价老用户不变(grandfather)。
2. 涨价搭配提额/加功能,渐进式。
3. 高级功能上移到更高档(触发升级弹窗)。
4. 隐性调整(减权益/提消耗而非明改价)——谨慎,信任层人设慎用。
5. 提前公告涨价,鼓励老用户锁价 = 一波转化。

## 七、多站/多产品 Stripe 架构
新建**账号**(信息+风险隔离)优先于同账号新建产品(违规连坐);商家主体信息可复用。详见 `payment-ops.md`。
