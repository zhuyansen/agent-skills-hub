# 通用 SEO 水系模型(做任何站都适用)

> 把 SEO 当水系:**先有源头才有水可流**。力气按"最不 AI"排序,不是按"最省事"排序。
> 源自哥飞会议的提法 + AgentSkillsHub 实战 —— **每个构件都是撞过的坑,不是教科书**(出处见文末伤疤索引)。

## 水系全景(十个构件,按水流排序)

| # | 构件 | 是什么 | 谁做 | 水位(各看一个数) |
|---|---|---|---|---|
| ①a | **进水口 · 传统权威** | 外链 · PR/行业媒体 · 真实社区 · 专家引用(do/nofollow 混合)。**外链两层论(07-23)**:**权重层** = 垂类付费客座博客(自定义锚文本,Adsy 自采,日 1-2 条封顶)+ 高 DR 平台页(Stripe 捐赠页/Notion 等);**发现层** = 免费目录/导航站(收录触发 + AI grounding 语料,不传权重,只验收不扩量) | 🧑 **人主导** | 权威 referring domains 数 |
| ①b | **进水口 · GEO/AI 引用** | 被 AI 搜索引用:数据集 · 论文 · llms.txt/skill.md · **MCP/CLI/GPTs 分发**(装进 agent = 不走搜索的常驻触点)。⚠️ GEO 侧"**被提及 > 被链接**":Ahrefs 7.5万品牌研究,AI 可见性相关性 = YouTube 提及 0.737 > 无链接品牌提及 0.66-0.71 >> 外链数 0.28 —— 弹药清单别只备链接型资产;传统位次仍是入场券(AI 联网搜的是前排结果,排名丢引用也丢) | 🤖备料 🧑按发送 | AI 引荐会话数 + 品牌被提及数 |
| ①c | **进水口 · 平台原生市场** | GitHub/npm/Chrome 商店等平台**内部搜索算法**(仓库名关键词 · GitHub tag 上限 20 个 · README 结构化 · npm keywords)—— 与 Web SEO 独立的第三类检索入口;我们有 CLI+MCP 包却从未针对性优化 | 🤖 AI | 平台内搜索排位/安装量 |
| ② | **水库 · 品牌实体** | 品牌词 · 实体图 sameAs(npm/HF/GitHub)· 域名信任 —— 权重先蓄进实体,管道才有得分发;品牌词终态验收 = SERP 出现 **sitelink**(出现后基本抢不走) | 🤖织图 🧑定名 | 品牌词位次 + CTR |
| ③ | **分层 · 站点架构** | hub → cluster → 子页 主题分层(让权重有路可走) | 🤖 AI | 一次定型(改架构成本随页数暴涨) |
| ④ | **流动 · 内链传权** | 权重从"拿到外链的页"流向"要冲排名的页" | 🤖 AI | 孤儿页数(**孤儿页 = 断管**) |
| ⑤ | **撑子页 · 程序化长尾** | 批量长尾页拿排名;**只建有曝光的词(pos 8–20)**,建前查重 | 🤖 卡阈值 | 机会词覆盖数 |
| ⑥ | **活水 · 内容流** | 带日期的内容流(日报/周榜/变更记录)让爬虫勤来,还吃时效词;**事件承接页**(新模型发布/skill 爆雷当天出专页,吃时效词,事后转 FAQ+订阅沉淀);纯数据页 = 死水潭。起量模式五分类可当选题参照:事件型/品牌词承接型/工具词型/fandom 圈层型/母体导流型 | 🤖 AI | 内容流更新频率 |
| ⑦ | **排水 · 索引卫生** | 低质页 noindex · canonical 去重 · 控爬虫预算 —— 垃圾页挤占收录配额 | 🤖 AI | 已收录/已提交比 |
| ⑧ | **水车+回流 · 转化与留存** | 埋点看"水做不做功"(注册/装机/付费);订阅/告警把流过的水收回水库 | 🤖埋 🧑看 | 转化事件/会话 · 订阅数 |
| ⑨ | **迭代 · 反馈闭环** | 日盘 cron + 周复盘 → 抓机会词/孤儿页 → 回"开哪个进水口/通哪根管" | 🤖 AI | 周复盘执行了没 |

## 施工顺序(冷启动的隐藏依赖)

**⑤ 的 "pos 8-20 才建页"阈值需要 GSC 数据 —— 新站第一天没有,这条规则失效。** 正确顺序:

1. **架构(③)先于一切**:管道先铺,别等 10 万页再动 URL 结构。
2. **进水口(①)从第 1 天持续做**,永远是主线,不是"内容做完再说"。
3. **撑子页冷启动 = 先播种后修剪**:没数据时按品类广度播少量 hub+子页,3-4 周 GSC 出曝光后,再按阈值扩产/砍掉。
   **冷启动选词双阈值(无 GSC 数据期的代理指标,多信源交叉验证)**:
   - **KGR = `allintitle:` 结果数 ÷ 月搜索量,< 0.25 可做**(变体:EKGR = allintitle×(1+KD/100)÷月搜索量;KDROI = ((日搜索量×0.1×365)÷(所需外链数×100)−1)×100%)
   - 或 SERP 结果数 < 100 + KD < 10(次选 < 30)
   - 有对标大站时的第三解:SEMrush 导竞品全量词库,聚焦其**排名 4-20 + 月搜索 1万-10万**的词(前 3 名词占 92% 流量打不过,4-20 区间才是新站锚定区)
4. **水位表第 1 天就记**:基线比绝对值重要。
5. **⚠️ 曝光 ≠ 排名**:3-4 周是"GSC 出曝光"的窗口;真拿到首页排名平均 ~180 天、1 年内仅 5.7% 新页做到(Ahrefs 实测)。别把出曝光误当排名成功,长周期检查点设 6 个月。
6. **零供给新词豁免条款**:pos 阈值只管存量词;全新词(SERP 供给≈0)唯一变量是速度 —— 官宣→上站→首批外链压进 3-7 天,我们赛道等价物 = 新工具发布 **48h 内出 skill+audit+场景页,建页 SLA 24h**。
7. **上线节奏规则(2026-03 核心更新后的保险)**:新页型 **> 1,000 页时分批入 sitemap**——首批 5-10% 验收录质量,GSC 无异常再放闸。"规模化内容滥用"已不区分是否 AI 生成,Best-X-for-Y 模板页是重灾区(71% 负面影响);哥飞量化红线"引用域<100 / DR<40 / 前20名页<100 = 新站,不该程序化批量发页"——我们(DR6/13万页)属于有意识的偏离(页面基于真实数据而非空模板),但新增页型必须分批,这是成本≈0 的保险。

## 🔑 水系铁律

1. **先接最难、最不 AI 的进水口(人),子页才有水流。**
2. **AI 火力别浪费在最省事的撑子页上** —— 铺再多管子也旱着。
3. **水要蓄得住(水库)、收得回(回流)、排得出(排污)**,否则灌再多也漏光。
4. **人只做 AI 做不了的**:选源、定名、按发送键、看数据拍板;其余全部 AI 化(AI 备料到只差发送)。

## 判断清单(接手任何站先问)

- **进水口开了吗?** 有几条真实高权威 dofollow 外链?没有 = 先开源头,别急着建页。
- **提交前查过对方收什么吗?** 每个收录平台有自己的"本体论"(只收 SaaS/只收 skill/只收基准),名字像目录 ≠ 是目录;通道(PR/issue/表单)开没开先探再动手。
- **AI 爬虫进得来吗?** robots.txt 别挡 GPTBot/ClaudeBot/PerplexityBot —— 挡了等于焊死 GEO 进水口;llms.txt / skill.md 给 agent 专用读法。
- **名字和域名自带信任吗?** 类目描述词当品牌名 + 低信任后缀(.top/.xyz)= 品牌词难排的两刀。新站避开;老站靠权重线(数据集/论文/PR)补。**折中标准(哥飞完整立场)**:自然可读的关键词短语域名 OK(CapitalizeMyTitle.com 式),硬拼堆砌不行(bestaixxx 式);单功能工具站"关键词即品牌名"更优,多品类枢纽站用品牌名。两刀里第一刀的本质是"品牌名与通用类目词语义重叠"——这解释了品牌词卡 pos 11 的一半原因。**例外**:零供给新词的 EMD(精确匹配域名)是最快承接方式,但那是即用即弃的套利站打法,别和主站品牌策略混用。**购前双查**:virustotal 查黑历史 + archive.org 查前世。
- **有孤儿页吗?** 抓代表页数内链:只往 home/pricing 导、不连 hub/兄弟页 = 断管,先通管。
- **撑子页卡阈值了吗?** pos 50+ 的词不建页(权威病,造页无效还稀释);只对 pos 8–20 建。**例外判别**:判死前先看一眼 SERP 供给质量——前十若多为论坛帖/过期站/视频凑数,那不是权威病而是"没人做对供给"(allintitle 供需比 <0.25 佐证),值得单独重做。老词也有机会:竞争者每年自然消亡,谷歌需要新供给替代旧劣质供给。
- **建页前查重了吗?** 先 grep 现有 slug + 看真实 PV 热门页 —— 两页抢同一批词 = 自相竞争(GSC 看不出来,要交叉流量数据)。
- **排水做了吗?** 低质页 noindex 了没?Search Console 看已收录/已提交比 —— 比例低 = 管道堵了,先清淤再灌水。
- **转化能看见吗?** 埋了事件没?看不见转化 = 只能盲目堆流量。
- **迭代接决策了吗?** digest 出来后,有没有每周"挑 1 实验 → 改 1 处 → 复测"的仪式?

## 最容易犯的错

新手和 AI 都爱从「撑子页」开工,因为最省事、最有成就感。
但那是**用错力气** —— 撑子页只是"接水的田",不是水。
**先把最难的进水口接通,子页才有水可流。**

## 伤疤索引(每条规则的实战出处)

- **水库两刀**:类目描述名 + `.top` 后缀 → 品牌词 pos 11;sameAs 绑 HF 数据集 = 双向实体确认
- **GEO**:GSC 实测 pos 6-9 却 0.1-0.4% CTR(AI Overviews 吃点击);Plausible 已现 ChatGPT/Perplexity 引荐
- **断管**:7,924 个 /audit/ 页曾是只有 6 条出链的孤儿岛,skill 页也不回链
- **自相竞争**:新建 ppt-skills 撞上已有 ppt-presentation,靠 Plausible PV 才发现,合并收场
- **排水**:130K 页只索引 stars≥50(就是排水闸);badge 曾 127K 文件撑爆 dist
- **活水**:日报每天产出却只发社媒,从未落成网页(错失的活水)
- **回流**:Clarity 实测高跳出、查完即走;"评级变化告警"是天然回访理由
- **checklist 首次 dogfood** 就抓到自己 llms.txt 404,当场补上
- **进水口三态**(07-10 清 17 站实证):自动过审型(LibHunt 单字段秒收)/ 被动爬取型(挂规范 SKILL.md 即被收录,不收提交)/ 关门型(wong2 PR+issue 双关"只出不进"、StackShare 本体论不符、iCloud 中继邮箱无路)—— 三态判型 5 分钟,判错浪费半天
- **实体双环**:Wikidata P856 指站 + 站内 sameAs 指回 QID = 双向确认;UI 陷阱是把属性名填进值框(P31=“instance of”),验证必须用 EntityData API 查 claims,不能只看页面
- **数据集一鱼三吃**:同一份 CSV → HF/Kaggle 数据集 + Zenodo DOI(10 分钟即"可引用文献",不等论文)+ Kaggle 分析 notebook + Dev.to 教程文,每个衍生品都是独立回链
- **开出去的 PR 也会烂**:4 月的 awesome PR 里数字停在 67K/117K,刷新到现值 + 补新定位才有 merge 相;提交前先 `gh pr list --author` 查重

## 适用边界(2026-07-23 大师库对照补充)

- **本框架假设你的赛道有结构性差异化空间**(数据集、技能、垂直深度)。在完全同质化、纯拼手速的红海品类(泛用型 H5 小游戏矩阵等),免费自然流量窗口期可能短到不值得投入,应及时转付费杠杆——那是另一种商业模式("流量即资产",终点是广告变现),我们学它的**入口发现和承接速度**,不学"流量即终点";哥飞"SEO 是放大器不是启动器"仍适用于我们的信任层 SaaS 模式。
- **分析单位要分清**:页面级扩产守曝光阈值(单站内堆同质页=稀释);站点级矩阵铺量是组合概率投资(每站独立域名、边际成本趋零时成立)——我们单域名权重资产站不采用后者,也别把后者的逻辑误用到前者。
- **谷歌"信任探针"机制**(权威病的机制解释):先给小流量测试你能不能接住,接住了才给更多,循环放大——这就是为什么 pos 50+ 的词改标题救不回来,只能走权重线。
